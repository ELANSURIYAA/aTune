Option 1 : Conceptual Data Model Development Pipeline

  1. Agents to consider : Fabric Model Conceptual → Fabric Model Data Constraints

  2. What we will get :

     • A complete conceptual data model for credit card reporting will be produced, covering all five required reports (Application Funnel, Campaign Performance, Credit Risk, Fraud Screening, and Transaction Analysis). The Fabric Model Conceptual agent will analyze the requirements document to identify and define all core business entities (Application, Campaign, Credit Risk, Fraud Screening, Transaction, Funnel Stage), their attributes, and the relationships between them, producing a conceptual entity-relationship diagram that serves as the foundation for all downstream modeling and reporting activities.

     • All business goals, KPIs, and use cases for credit card reporting will be explicitly documented and validated against the requirements. The agent will extract the reporting objectives, identify the key performance indicators required for each report, and map them to the conceptual entities and attributes, ensuring that the model supports all stated business objectives with full traceability from requirement to entity to report.

     • Data constraints, business rules, and validation logic will be formally defined and documented for every entity and attribute in the conceptual model. The Fabric Model Data Constraints agent will extract and codify all business rules (e.g., calculation rules for funnel conversion rates, credit risk scoring logic, fraud detection thresholds, transaction categorization rules), data retention policies, audit trail requirements, and security access rules, ensuring that the conceptual model is not only structurally complete but also operationally enforceable and compliant with regulatory and governance standards.

     • Explicit domain-bounded context modeling and formal subject-area catalogues will not be produced by the available agents. While the Fabric Model Conceptual agent identifies entities and relationships, it does not perform enterprise-level domain modeling or produce a structured subject-area taxonomy that organizes entities into business domains (e.g., Customer Domain, Product Domain, Transaction Domain). This gap means that the conceptual model may lack the higher-level organizational structure required for enterprise data architecture alignment and cross-domain integration.

     • A formal business glossary with standardized definitions, synonyms, and data stewardship assignments will not be generated. The available agents produce entity and attribute definitions within the context of the conceptual model, but they do not create a standalone, enterprise-wide business glossary that can be used across multiple projects, systems, and reporting initiatives. This limits the reusability and consistency of terminology across the organization and may lead to ambiguity when the model is handed off to downstream teams or integrated with other data assets.

     • The workflow does not include explicit validation of the conceptual model against enterprise data governance standards, existing data architecture patterns, or integration with a centralized data governance platform. While the Fabric Model Data Constraints agent documents business rules and compliance requirements, there is no agent in the available set that performs formal integration with a data governance system, validates the model against enterprise naming conventions, or ensures alignment with existing logical and physical data models. This gap increases the risk of model rejection during governance review or rework during logical and physical modeling phases.

SECTION 4 — IMPROVEMENTS & OPTIMISATIONS

ACCURACY — How well the agent produces correct, on-target outputs and avoids errors.

| Criteria            | Description                                  |
|---------------------|----------------------------------------------|
| Self-validation     | The Fabric Model Conceptual agent must perform a self-validation check before finalizing the conceptual ER model by cross-referencing every entity, attribute, and relationship in the model against the requirements document to ensure 100% coverage of all stated business goals, KPIs, and use cases. The agent must produce a coverage matrix that maps each requirement to the corresponding model element and flag any unmapped requirements for escalation. The Fabric Model Data Constraints agent must validate that every business rule, constraint, and compliance requirement extracted from the requirements document is internally consistent (no conflicting rules), complete (all required rules are documented), and traceable to the conceptual entities and attributes. Any rule conflicts, missing rules, or untraceable rules must be flagged and escalated before the agent completes its execution. The Data Governance Integration Agent [NEW] must validate that the conceptual model passes all enterprise governance checks (naming conventions, data quality standards, compliance requirements) and that all validation results are documented in the governance approval report before handoff is approved. |
| Boundary definition | The Fabric Model Conceptual agent must NOT generate entities, attributes, or relationships that are not explicitly stated or reasonably inferred from the requirements document. The agent must NOT perform logical or physical modeling tasks (e.g., defining primary keys, foreign keys, data types, or normalization) — these are out of scope for conceptual modeling. The agent must NOT make assumptions about business logic or reporting requirements that are not documented in the requirements. The Fabric Model Data Constraints agent must NOT invent business rules, constraints, or compliance requirements that are not stated in the requirements document. The agent must NOT modify or interpret business rules — it must extract and document them exactly as stated. The agent must NOT approve incomplete or ambiguous rule sets — all ambiguities must be escalated. The Domain & Subject Area Architect agent [NEW] must NOT create domain boundaries or subject areas that conflict with existing enterprise architecture standards or that are not supported by the conceptual entities in the model. The Business Glossary Manager agent [NEW] must NOT create glossary definitions that contradict existing enterprise glossary terms or that are not traceable to the conceptual model entities and attributes. |
| Known issues log    | The Fabric Model Conceptual agent must log all instances where entity definitions are ambiguous, relationships are unclear, or attributes are missing from the requirements document. The agent must log all cases where multiple entities could represent the same business concept (e.g., "Customer" vs. "Cardholder") and document the rationale for the chosen entity name. The agent must log all inferred relationships (relationships not explicitly stated in the requirements but reasonably inferred from business logic) and flag them for stakeholder validation. The Fabric Model Data Constraints agent must log all instances where business rules conflict with each other, where compliance requirements are incomplete or ambiguous, or where data retention policies are not clearly specified. The agent must log all cases where a business rule cannot be traced to a specific entity or attribute in the conceptual model, indicating a potential gap in the model or the requirements. The Data Governance Integration Agent [NEW] must log all governance validation failures, naming convention violations, and compliance check failures, along with the specific policy or standard that was violated and the recommended remediation action. |

COST EFFECTIVE — How to minimise cost per agent without sacrificing quality.

| Criteria                       | Description               |
|--------------------------------|---------------------------|
| Low cost model routing         | Route the Fabric Model Data Constraints agent to Claude Sonnet 3.5 instead of GPT-4o for all rule extraction, compliance documentation, and constraint validation subtasks. These subtasks are primarily rule-based and do not require the advanced reasoning capabilities of GPT-4o, and Claude Sonnet 3.5 provides comparable accuracy at a lower cost per token. Reserve GPT-4o only for the Fabric Model Conceptual agent, where complex entity extraction, relationship mapping, and conceptual reasoning are required to produce an accurate, complete conceptual ER model. Route the Business Glossary Manager agent [NEW] to Claude Sonnet 3.5 or GPT-3.5 Turbo for glossary definition generation, synonym management, and data stewardship assignment, as these are primarily linguistic and definitional tasks that do not require frontier model capabilities. Reserve GPT-4o for the Domain & Subject Area Architect agent [NEW] and the Data Governance Integration Agent [NEW], where enterprise-level reasoning, complex validation logic, and multi-dimensional cross-referencing are required. |
| Token size reduction           | Cache the requirements document and pass only the relevant sections to each agent based on its subtask. For example, the Fabric Model Conceptual agent only needs the entity and relationship sections of the requirements document for entity extraction, not the full document including compliance and audit sections. This reduces input token count by approximately 40-50% per agent call. Summarize the output of the Fabric Model Conceptual agent before passing it to the Fabric Model Data Constraints agent — instead of passing the full conceptual ER model with all entity definitions, attributes, and relationships, pass only the entity and attribute names and IDs that the constraints agent needs to trace rules and constraints. This reduces input tokens for the constraints agent by approximately 60%. Use prompt caching for the enterprise data architecture standards, naming conventions, and compliance requirements KB when calling the Data Governance Integration Agent [NEW] — these reference documents are static and can be cached across multiple agent calls, reducing input token costs by approximately 30% for governance validation tasks. |
| Hybrid model                   | Handle all duplicate detection, null value checks, and format validation in the Fabric Model Data Constraints agent using deterministic regex or rule-based logic outside the LLM loop. For example, checking that all entity names follow a specific naming convention (e.g., PascalCase, no special characters) can be done with a simple regex check rather than an LLM call, eliminating unnecessary token usage for trivial validation tasks. Use rule-based logic to generate the coverage matrix in the Fabric Model Conceptual agent — once the agent has produced the list of entities, attributes, and relationships, a deterministic script can cross-reference each model element against the requirements document to produce the coverage matrix without requiring an additional LLM call. This reduces token usage by approximately 20% for the conceptual agent. Use deterministic API calls to integrate the conceptual model metadata into the data governance platform in the Data Governance Integration Agent [NEW] — the metadata ingestion, entity registration, and approval tracking steps do not require LLM reasoning and can be handled by direct API calls to the governance platform, eliminating unnecessary LLM token usage for integration tasks. Reserve LLM calls only for the validation logic and governance approval report generation, where reasoning and natural language output are required. |
