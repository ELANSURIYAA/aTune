ACCURACY — How well the agent produces correct, on-target
outputs and avoids errors.

| Criteria            | Description                          |
|---------------------|--------------------------------------|
| Self-validation     | Requirements Parsing & Validation Agent must cross-reference every parsed section against a master requirements template to confirm that all mandatory sections (business goals, entities, attributes, relationships, business rules, compliance, audit) are present and complete before marking the document as validated; Fabric Model Conceptual agent must validate that every entity has at least one primary key, all attributes are assigned to an entity, and all relationships have defined cardinality before producing the ER diagram; Fabric Model Data Constraints agent must validate that every business rule references a specific entity and attribute, all compliance requirements are mapped to enforceable constraints, and no rule conflicts with another before finalizing the constraints catalog; Model Synthesis & Integration Agent must validate that the assembled model covers 100% of the requirements, all entities from prior stages are included, and the governance validation API returns a pass status before generating the handoff package. |
| Boundary definition | Requirements Parsing & Validation Agent must NOT alter, paraphrase, or invent any requirement text; must NOT proceed if any critical section (entities, compliance, audit) is missing; must NOT resolve ambiguities without stakeholder input. Fabric Model Conceptual agent must NOT create entities that are not explicitly stated or clearly implied in the requirements; must NOT define relationships without evidence from the requirements or business process documentation; must NOT proceed to relationship modeling if any core entity is ambiguous or incomplete. Fabric Model Data Constraints agent must NOT invent business rules or compliance requirements; must NOT approve constraints that conflict with regulatory standards; must NOT omit mandatory audit or security requirements. Model Synthesis & Integration Agent must NOT hand off an incomplete model; must NOT bypass governance validation; must NOT proceed if any prior agent output is missing or failed validation. |
| Known issues log    | Requirements Parsing & Validation Agent should log and avoid: (1) false positives in gap detection where optional sections are flagged as missing, (2) failure to detect implicit requirements buried in narrative text, (3) over-escalation of minor ambiguities that could be resolved through standard interpretation rules. Fabric Model Conceptual agent should log and avoid: (1) creating duplicate entities with slightly different names for the same business concept, (2) defining many-to-many relationships without identifying the associative entity, (3) omitting subtype/supertype relationships when entity hierarchies are implied in requirements. Fabric Model Data Constraints agent should log and avoid: (1) defining constraints that are too restrictive and block valid business scenarios, (2) missing cascading delete or update rules for dependent entities, (3) failing to flag PII attributes that require encryption or masking. Model Synthesis & Integration Agent should log and avoid: (1) integration failures due to API timeouts or governance system unavailability without retry logic, (2) incomplete handoff packages missing audit logs or validation reports, (3) failure to reconcile conflicting entity definitions from multiple prior agent outputs. |

sacrificing quality.

| Criteria                       | Description               |
|--------------------------------|---------------------------|
| Low cost model routing         | Requirements Parsing & Validation Agent: Use GPT-4o-mini (~$0.00015 per 1k input tokens) for initial document section extraction and template matching (low complexity, rule-based), then route to GPT-4o (~$0.005 per 1k input tokens) only for gap detection and ambiguity analysis where reasoning is required. Fabric Model Conceptual agent: Use GPT-4o-mini for attribute extraction and data type assignment (structured, low complexity), then route to Claude Sonnet 3.5 (~$0.003 per 1k input tokens) for entity relationship modeling and ER diagram generation where deep reasoning and pattern recognition are required. Fabric Model Data Constraints agent: Use GPT-4o-mini for rule extraction and constraint formatting (structured, template-based), then route to GPT-4o only for compliance validation and conflict resolution where regulatory reasoning is required. Model Synthesis & Integration Agent: Use GPT-4o-mini for artifact collection and checklist validation (low complexity), then route to Claude Sonnet 3.5 for governance integration and final model validation where comprehensive cross-validation reasoning is required. |
| Token size reduction           | Requirements Parsing & Validation Agent: Summarise or cache the requirements template (which is static across all runs) to avoid re-sending it in every API call; extract only the relevant sections for gap detection rather than re-processing the entire document in each validation step; use document chunking to process large requirements documents in sections rather than as a single context. Fabric Model Conceptual agent: Cache the business process documentation and data modeling standards (static knowledge base content) to avoid re-sending in every call; pass only the parsed requirements and entity candidates to the relationship modeling step rather than the full requirements document; use entity-by-entity processing for large models rather than attempting to model all entities in a single call. Fabric Model Data Constraints agent: Cache the compliance standards and regulatory frameworks (static content) to avoid re-sending in every call; pass only the relevant entities and attributes to the rule extraction step rather than the full conceptual model; use rule-by-rule processing for large rule sets rather than processing all rules in a single call. Model Synthesis & Integration Agent: Cache the governance validation schema and handoff package template (static content); pass only the delta or new entities to the governance API rather than re-validating the entire model in each call; use incremental validation rather than full model re-validation for minor updates. |
| Hybrid model                   | Requirements Parsing & Validation Agent: Use regex and keyword matching (non-agentic) to identify section headers and extract structured fields (e.g., entity names, attribute lists) before invoking the LLM for gap detection and ambiguity analysis; use rule-based logic to classify gap severity (critical if compliance/audit missing, high if entities missing, medium if attributes incomplete) without LLM calls. Fabric Model Conceptual agent: Use rule-based logic (non-agentic) to validate that every entity has a primary key, all attributes have data types, and all relationships have cardinality defined before invoking the LLM for relationship reasoning and ER diagram generation; use deterministic cardinality inference rules (e.g., "each X has many Y" → one-to-many) to reduce LLM reasoning overhead. Fabric Model Data Constraints agent: Use rule-based lookups (non-agentic) to flag known PII attribute names (e.g., SSN, credit_card_number, email) and apply standard encryption requirements without LLM calls; use deterministic compliance mapping (e.g., if entity contains PII → apply GDPR retention rules) to reduce reasoning overhead. Model Synthesis & Integration Agent: Use deterministic checklist validation (non-agentic) to confirm that all required artifacts (ER diagram, rules catalog, validation report, audit log) are present before invoking the LLM for governance integration and final validation; use rule-based coverage calculation (count of requirements vs. count of entities/attributes in model) to determine completeness without LLM reasoning. |
