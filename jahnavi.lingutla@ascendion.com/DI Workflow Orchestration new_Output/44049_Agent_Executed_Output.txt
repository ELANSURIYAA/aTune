While the Fabric Model Conceptual agent lists entities, attributes, and common data elements, it does not generate a structured glossary artifact that can be published to enterprise metadata repositories or used for organization-wide data literacy and semantic consistency.

COST EFFECTIVE — How to minimise cost per agent without sacrificing quality.

| Criteria                       | Description                        |
|--------------------------------|---------------------------------------|
| Low cost model routing         | Route the Business Goals & Use Case Validation Agent [NEW] to GPT-4o only for stakeholder input synthesis and conflict resolution; use GPT-3.5 Turbo or Claude Haiku for extracting goals and KPIs from the requirements document, as this is a straightforward extraction task. Route the Domain & Subject Area Modeling Agent [NEW] to Claude Sonnet for domain boundary definition and subject-area hierarchy creation, as this requires structured reasoning with low hallucination risk; use GPT-3.5 Turbo for cross-checking against the enterprise taxonomy, as this is a lookup and comparison task. Route the Business Glossary Agent [NEW] to GPT-4o only for defining new terms and resolving conflicts; use GPT-3.5 Turbo for extracting terms from requirements and cross-checking against the existing glossary, as these are straightforward lookup and extraction tasks. Route the Fabric Model Conceptual agent to GPT-4o for entity and relationship extraction from complex requirements, as this requires strong reasoning; use Claude Haiku for attribute definition and ER diagram formatting, as these are more structured tasks. Route the Fabric Model Data Constraints agent to Claude Sonnet for business rule extraction and validation logic definition, as this requires logical reasoning with low error tolerance; use GPT-3.5 Turbo for rule catalog formatting and traceability documentation. Route the Model Integration & Handoff Agent [NEW] to GPT-4o only for gap analysis and validation report generation; use GPT-3.5 Turbo for integration API calls, sign-off tracking, and handoff package assembly, as these are structured, deterministic tasks. |
| Token size reduction           | Cache the requirements document and pass only the relevant sections to each agent based on its scope — the Business Goals & Use Case Validation Agent [NEW] only needs the business goals and KPI sections, not the full technical specifications, reducing input tokens by ~40%. Cache the enterprise taxonomy and glossary and pass only the relevant domain or term subset to the Domain & Subject Area Modeling Agent [NEW] and Business Glossary Agent [NEW], reducing input tokens by ~50%. Summarize the output of the Fabric Model Conceptual agent before passing it to the Fabric Model Data Constraints agent — pass only the entity and attribute list, not the full ER diagram metadata, reducing input tokens by ~30%. Cache the conceptual model template and pass only the variable data sections (entities, attributes, relationships, rules) to the Model Integration & Handoff Agent [NEW], reducing input tokens by ~35%. Use structured JSON outputs instead of verbose narrative documentation across all agents, reducing output tokens by ~40% while maintaining completeness and traceability. |
| Hybrid model                   | Handle requirements document parsing and section extraction using regex or NLP libraries (e.g., spaCy, NLTK) outside the LLM loop before passing to the Business Goals & Use Case Validation Agent [NEW], avoiding unnecessary LLM calls for straightforward text extraction. Use rule-based lookup and comparison logic to cross-check domain candidates and glossary terms against the enterprise taxonomy and existing glossary in the Domain & Subject Area Modeling Agent [NEW] and Business Glossary Agent [NEW], avoiding LLM calls for deterministic matching tasks. Handle entity and attribute extraction from structured requirements sections (e.g., tables, lists) using pandas or regex in the Fabric Model Conceptual agent, reserving LLM calls only for ambiguous or narrative-heavy sections. Use rule-based validation logic (e.g., regex for data type validation, SQL for constraint checking) in the Fabric Model Data Constraints agent to validate business rules before passing to the LLM for documentation and traceability, reducing LLM calls by ~30%. Handle model completeness checks (e.g., counting entities, verifying relationship completeness) using deterministic scripts in the Model Integration & Handoff Agent [NEW], reserving LLM calls only for gap analysis and validation report narrative generation. |
