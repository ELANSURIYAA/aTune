SECTION 4 — IMPROVEMENTS & OPTIMISATIONS

1. Efficiency Gains

   The Fabric Model Conceptual and Fabric Model Data Constraints agents can be run as a single chained call with intermediate state passed directly in memory rather than serializing and deserializing the conceptual model between steps, eliminating a full API round-trip and reducing pipeline latency by 20–30%. The conceptual data model diagram template (entity-relationship diagram layout, styling, legend) should be cached and reused across multiple credit card reporting projects rather than regenerated each time, since the visual format and structure remain consistent even as entity details change, saving rendering time and API tokens. The Fabric Conceptual Model Stakeholder Validator agent should batch all stakeholder notification emails and calendar invites for review sessions into a single API call to the collaboration platform rather than sending individual messages per stakeholder, significantly reducing email service overhead and avoiding rate-limit throttling on large stakeholder lists. The requirements-to-entity traceability matrix generation should be performed incrementally as entities are defined rather than as a separate post-processing step, so that the matrix is always up-to-date and does not require a full re-scan of the conceptual model at validation time, reducing compute cost and latency.

2. Agent Accuracy Improvements

   Fabric Model Conceptual agent: Prompt engineering should include explicit examples of well-formed entity definitions, attribute lists, and relationship descriptions specific to the credit card domain (e.g., how a Transaction relates to both a Credit Card Account and a Merchant, how Statement aggregates Transactions over a period), reducing ambiguity and ensuring consistent output structure across runs. A post-generation validation step should confirm that every entity has at least one primary key attribute candidate, that all relationships specify cardinality (one-to-many, many-to-many), and that no orphaned entities exist without any relationships, catching structural errors before the model proceeds to constraint definition. Structured output formatting (JSON schema with required fields for entity name, description, attributes array, relationships array) should be enforced on the agent's response to prevent narrative drift, missing fields, or inconsistent formatting that would break downstream processing.
   
   Fabric Model Data Constraints agent: The agent should be prompted to explicitly cite the source requirement or business rule text that justifies each constraint it defines, creating a direct traceability link from constraint back to requirement and reducing the risk of hallucinated or inferred rules that were not actually stated in the requirements document. A cross-validation step should check that every constraint references only entities and attributes that were actually defined by the Fabric Model Conceptual agent, flagging any orphaned or misnamed references that would indicate a hallucination or parsing error. Confidence scoring should be applied to each business rule, with rules derived from explicit requirement statements scored high and rules inferred from implicit patterns scored lower and flagged for human review before being included in the final model.
   
   Fabric Conceptual Model Stakeholder Validator agent [NEW]: The agent should enforce a structured feedback template for stakeholders (e.g., "Entity X is missing attribute Y because requirement Z requires it" rather than free-form comments) to ensure feedback is actionable, traceable, and can be automatically linked back to specific model elements and requirements. A completeness check should verify that every core business domain identified in the requirements (customer, account, transaction, limits, statements, risk) has been reviewed and approved by at least one subject matter expert from that domain, preventing gaps where a domain is overlooked or under-represented in the validation process. The agent should automatically detect conflicting feedback (e.g., Stakeholder A says attribute X is required, Stakeholder B says it is out of scope) and escalate conflicts to a designated decision-maker rather than silently choosing one opinion over the other, ensuring that all conflicts are resolved with clear rationale and audit trail.

3. Cost Reduction

   The Fabric Model Conceptual agent should use a tiered model strategy: GPT-4o for the initial requirements analysis and entity derivation (complex reasoning required), then switch to GPT-3.5 Turbo or Claude Haiku for generating the final formatted output artifacts (entity list, attribute tables, diagram annotations) once the conceptual structure is determined, reducing per-run cost by 40–60% on the output generation phase without sacrificing analytical quality. The Fabric Model Data Constraints agent should cache common business rule patterns for credit card reporting (e.g., "account status must be one of [Active, Closed, Suspended]", "transaction amount must be non-negative", "statement period must not overlap") and reuse them across multiple projects rather than re-deriving them from scratch each time, reducing input token size and API calls. The Fabric Silver Model Reviewer agent (when used) should perform validation checks in a single pass rather than making multiple API calls per entity or attribute, batching all validation logic into one prompt with structured output (JSON array of findings) to minimize API overhead. The Fabric Conceptual Model Stakeholder Validator agent [NEW] should use a lightweight model (GPT-3.5 Turbo or Claude Haiku) for routine tasks like formatting feedback logs, generating email notifications, and updating traceability matrices, reserving GPT-4o only for complex tasks like resolving conflicting stakeholder comments or synthesizing approval summaries, reducing average per-run cost by 50–70%. The full pipeline should be scheduled to run during off-peak hours (e.g., overnight or weekends) when API latency is lower and rate limits are less likely to be hit, avoiding costly retries and timeouts that can occur during peak business hours when many concurrent workflows are executing.
