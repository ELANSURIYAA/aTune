Overview of Program\n\nThe bronze.usp_Load_bronze_Layer_Full stored procedure serves as the master orchestration procedure for executing a comprehensive full refresh ETL pipeline across all bronze layer tables in the data warehouse architecture. This procedure is designed to coordinate the sequential loading of twelve distinct bronze layer tables from their respective source systems, ensuring a complete and synchronized data refresh across the entire bronze layer. The primary purpose of this implementation is to provide a centralized, automated mechanism for refreshing all bronze layer tables in a single execution, thereby simplifying the ETL workflow and reducing manual intervention. The business context of this procedure revolves around maintaining an up-to-date bronze layer that serves as the foundational data repository for downstream silver and gold layer transformations, enabling accurate reporting and analytics. The business value delivered by this orchestration procedure includes improved operational efficiency through automation, reduced risk of human error in manual table loading processes, comprehensive audit trail generation for compliance and troublesability, and consistent execution patterns across all bronze layer table loads. The procedure begins by declaring and initializing multiple variables to track execution metrics including procedure name, start time, end time, execution duration, overall status, error details, row counts, table counts, current user, and a unique batch identifier. The batch identifier is generated using the start timestamp formatted as 'YYYY-MM-DD HH24:MI:SS.FF3' to ensure uniqueness and traceability across execution runs. The procedure logs the start of the overall ETL pipeline process using DBMS_OUTPUT statements that display the start timestamp, executing user, and batch identifier in a formatted banner for easy identification in logs. The core processing logic involves sequential invocation of twelve individual table-specific loading procedures, each responsible for loading data into a specific bronze layer table. The first table load invokes bronze.usp_Load_bz_New_Monthly_HC_Report with the batch identifier as a parameter to load the bz_New_Monthly_HC_Report table. The second table load invokes bronze.usp_Load_bz_SchTask with the batch identifier to load the bz_SchTask table, which notably excludes TIMESTAMP columns during the load process. The third table load invokes bronze.usp_Load_bz_Hiring_Initiator_Project_Info with the batch identifier to load the bz_Hiring_Initiator_Project_Info table. The fourth table load invokes bronze.usp_Load_bz_Timesheet_New with the batch identifier to load the bz_Timesheet_New table. The fifth table load invokes bronze.usp_Load_bz_report_392_all with the batch identifier to load the bz_report_392_all table. The sixth table load invokes bronze.usp_Load_bz_vw_billing_timesheet_daywise_ne with the batch identifier to load the bz_vw_billing_timesheet_daywise_ne table. The seventh table load invokes bronze.usp_Load_bz_vw_consultant_timesheet_daywise with the batch identifier to load the bz_vw_consultant_timesheet_daywise table. The eighth table load invokes bronze.usp_Load_bz_DimDate with the batch identifier to load the bz_DimDate table. The ninth table load invokes bronze.usp_Load_bz_holidays_Mexico with the batch identifier to load the bz_holidays_Mexico table. The tenth table load invokes bronze.usp_Load_bz_holidays_Canada with the batch identifier to load the bz_holidays_Canada table. The eleventh table load invokes bronze.usp_Load_bz_holidays with the batch identifier to load the bz_holidays table. The twelfth and final table load invokes bronze.usp_Load_bz_holidays_India with the batch identifier to load the bz_holidays_India table. After all twelve table loading procedures have been invoked, the orchestration procedure increments a counter variable v_TablesProcessed after each procedure call to track the total number of tables processed during the execution. The orchestration logic follows a sequential execution pattern where each table load is executed one after another without parallel processing, ensuring predictable execution order and simplified error tracking. Following the completion of all table loads, the procedure performs summary statistics calculation by querying the bronze.bz_Audit_Log table to aggregate execution metrics across all individual table loads for the current batch. The first aggregation query counts the number of successfully loaded tables and sums the total records inserted by filtering the audit log for records with the current batch identifier and a status of 'SUCCESS', storing results in v_TablesSucceeded and v_TotalRowsInserted variables. The second aggregation query counts the number of failed table loads and sums the total records that failed by filtering the audit log for records with the current batch identifier and a status of 'FAILED', storing results in v_TablesFailed and v_TotalRowsFailed variables. The total rows processed is calculated by summing the total rows inserted and total rows failed to provide a complete picture of data volume processed during the execution. The procedure calculates the total execution time by capturing the end timestamp using SYSTIMESTAMP and computing the difference between end time and start time, extracting days, hours, minutes, and seconds components and converting them to total seconds for precise duration measurement. The audit tracking mechanism relies on each individual table loading procedure inserting records into the bronze.bz_Audit_Log table with execution details, which are then aggregated by this master orchestration procedure to provide overall pipeline metrics. The procedure logs completion details using DBMS_OUTPUT statements that display end time, execution duration in seconds, total tables processed, tables succeeded, tables failed, total rows inserted, and total rows failed in a formatted banner for easy readability. After logging completion details, the procedure inserts a master audit record into the bronze.bz_Audit_Log table that captures the overall pipeline execution metrics including source pattern 'source_layer.*', target pattern 'bronze.*', load timestamp, start timestamp, end timestamp, processing user, processing time, overall status, total records processed, total records inserted, total records failed, batch identifier, and load type 'FULL_REFRESH_ALL_TABLES'. The procedure commits all audit log insertions to ensure persistence of execution metrics even if downstream processes encounter issues. The error handling approach is implemented through a comprehensive EXCEPTION block that captures any errors occurring during the execution of the orchestration procedure or any of the invoked table loading procedures. When an exception occurs, the error handling logic captures error details including error message using SQLERRM, error number using SQLCODE, and initializes error severity, error state, and error line variables to zero since Oracle does not provide direct equivalents to SQL Server's error metadata. The error handler calculates the execution time up to the point of failure and sets the overall status to 'FAILED' to indicate unsuccessful execution. The error handling mechanism logs comprehensive error details using DBMS_OUTPUT statements that display error number, error severity, error state, error line, and error message in a formatted error banner for troubleshooting purposes. The error handler inserts an error audit record into the bronze.bz_Audit_Log table with execution details and error information, ensuring that failed executions are tracked in the audit log alongside successful executions. After inserting the error audit record, the procedure commits the audit log entry to ensure the error is persisted for analysis and troubleshooting. The error handling concludes by re-throwing the error using RAISE_APPLICATION_ERROR with error code -20001 and the original error message, ensuring that calling processes are aware of the failure and can take appropriate action. The batch processing details involve using a unique batch identifier passed to all twelve table loading procedures, enabling correlation of all table loads within a single execution run and facilitating batch-level reporting and analysis. The execution metrics captured include start time, end time, execution duration, tables processed count, tables succeeded count, tables failed count, total rows processed, total rows inserted, total rows failed, processing user, and batch identifier, providing comprehensive visibility into pipeline performance. The reporting benefits include the ability to query the bronze.bz_Audit_Log table to analyze historical execution patterns, identify performance trends, detect recurring failures, and generate operational reports on ETL pipeline health and efficiency. The downstream impact of this orchestration procedure is significant as it ensures that all bronze layer tables are refreshed in a coordinated manner, providing a consistent and reliable foundation for silver and gold layer transformations that depend on bronze layer data for business intelligence and analytics workloads.\n\n# 2.
