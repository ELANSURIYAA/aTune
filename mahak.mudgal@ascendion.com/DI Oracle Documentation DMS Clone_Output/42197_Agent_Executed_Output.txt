# Oracle Stored Procedure Documentation - DMS Clone

## Procedure Information

**Procedure Name:** bronze.usp_Load_bronze_Layer_Full

**Schema:** bronze

**Type:** Stored Procedure

**Purpose:** Orchestrates the complete ETL pipeline for loading all bronze layer tables with full refresh

---

## Procedure Overview

This stored procedure serves as the master orchestrator for the bronze layer ETL pipeline. It executes multiple child procedures to load data from source layer to bronze layer tables, maintains comprehensive audit logging, and provides detailed execution statistics.

---

**Input Parameters:** None

**Output Parameters:** None

## Processing Logic

### 1. Initialization Phase

- Generate unique Batch ID using timestamp format: 'YYYY-MM-DD HH24:MI:SS.FF3'
- Log pipeline start information including:
  - Start timestamp
  - Executing user
  - Batch ID

### 2. Data Loading Phase

The procedure sequentially calls 12 child procedures to load bronze layer tables:

| Sequence | Procedure Name | Target Table | Notes |
|----------|---------------|--------------|-------|
| 1 | bronze.usp_Load_bz_New_Monthly_HC_Report | bz_New_Monthly_HC_Report | - |
| 2 | bronze.usp_Load_bz_SchTask | bz_SchTask | Excludes TIMESTAMP column |
| 3 | bronze.usp_Load_bz_Hiring_Initiator_Project_Info | bz_Hiring_Initiator_Project_Info | - |
| 4 | bronze.usp_Load_bz_Timesheet_New | bz_Timesheet_New | - |
| 5 | bronze.usp_Load_bz_report_392_all | bz_report_392_all | - |
| 6 | bronze.usp_Load_bz_vw_billing_timesheet_daywise_ne | bz_vw_billing_timesheet_daywise_ne | - |
| 7 | bronze.usp_Load_bz_vw_consultant_timesheet_daywise | bz_vw_consultant_timesheet_daywise | - |
| 8 | bronze.usp_Load_bz_DimDate | bz_DimDate | - |
| 9 | bronze.usp_Load_bz_holidays_Mexico | bz_holidays_Mexico | - |
| 10 | bronze.usp_Load_bz_holidays_Canada | bz_holidays_Canada | - |
| 11 | bronze.usp_Load_bz_holidays | bz_holidays | - |
| 12 | bronze.usp_Load_bz_holidays_India | bz_holidays_India | - |

Each child procedure receives the Batch ID as a parameter for audit tracking.

### 3. Statistics Aggregation Phase

Queries the audit log table (bronze.bz_Audit_Log) to calculate:

**Success Metrics:**
- Count of successfully loaded tables
- Total rows inserted across all successful loads

**Failure Metrics:**
- Count of failed table loads
- Total rows that failed to load

**Overall Metrics:**
- Total rows processed (inserted + failed)

### 4. Completion Phase

- Calculate total execution time using timestamp arithmetic
- Log comprehensive execution summary including:
  - End timestamp
  - Execution time in seconds
  - Tables processed/succeeded/failed counts
  - Row insertion/failure statistics

### 5. Audit Logging Phase

Insert master audit record into bronze.bz_Audit_Log with:
- Source: 'source_layer.*'
- Target: 'bronze.*'
- Load type: 'FULL_REFRESH_ALL_TABLES'
- All execution metrics and statistics
- Batch ID for traceability

### 6. Transaction Commit

Commit all changes to persist audit records.

## Error Handling

### Exception Block

When any error occurs during execution:

1. **Capture Error Details:**
   - Error message (SQLERRM)
   - Error number (SQLCODE)
   - Set error severity, state, and line (initialized to 0)

2. **Calculate Execution Metrics:**
   - End timestamp
   - Execution time up to failure point
   - Set overall status to 'FAILED'

3. **Log Error Information:**
   - Error number
   - Error severity
   - Error state
   - Error line
   - Error message

4. **Insert Error Audit Record:**
   - Record failure in bronze.bz_Audit_Log
   - Include error details in error_message field
   - Mark status as 'FAILED'

5. **Commit and Re-throw:**
   - Commit audit record
   - Re-throw error using RAISE_APPLICATION_ERROR with error code -20001

## Performance Considerations

1. **Sequential Processing:** Tables are loaded sequentially, not in parallel
2. **Single Transaction:** All audit logging is committed at the end
3. **Error Isolation:** Each child procedure should handle its own errors to allow pipeline continuation
4. **Audit Overhead:** Multiple queries to audit log table for statistics aggregation

## Migration Notes for DMS

### Oracle to Target Platform Considerations

1. **SYSTIMESTAMP Function:**
   - Oracle-specific function
   - Map to equivalent in target platform (e.g., GETDATE() in SQL Server, CURRENT_TIMESTAMP in PostgreSQL)

2. **TIMESTAMP Data Type:**
   - Ensure target platform supports equivalent precision
   - May need to adjust to DATETIME2 (SQL Server) or TIMESTAMP (PostgreSQL)

3. **DBMS_OUTPUT.PUT_LINE:**
   - Oracle-specific logging
   - Replace with target platform logging mechanism (e.g., PRINT in SQL Server, RAISE NOTICE in PostgreSQL)

4. **USER Function:**
   - Map to CURRENT_USER or SESSION_USER in target platform

5. **EXTRACT Function:**
   - Used for timestamp arithmetic
   - May need to use DATEDIFF or equivalent in target platform

6. **RAISE_APPLICATION_ERROR:**
   - Oracle-specific error handling
   - Map to THROW (SQL Server) or RAISE EXCEPTION (PostgreSQL)

7. **VARCHAR2 Data Type:**
   - Change to VARCHAR in most target platforms

8. **NVL Function:**
   - Replace with ISNULL (SQL Server) or COALESCE (standard SQL)

9. **Procedure Syntax:**
   - Adjust CREATE OR REPLACE syntax to target platform equivalent
   - Modify AS/IS keywords as needed

10. **Transaction Control:**
    - Ensure COMMIT behavior is consistent in target platform

## Monitoring and Troubleshooting

### Success Verification

Query the audit log to verify successful execution:

```sql
SELECT 
    batch_id,
    status,
    records_inserted,
    records_failed,
    processing_time,
    start_timestamp,
    end_timestamp
FROM bronze.bz_Audit_Log
WHERE load_type = 'FULL_REFRESH_ALL_TABLES'
ORDER BY start_timestamp DESC;
```

### Failure Investigation

Query failed table loads within a batch:

```sql
SELECT 
    target_table,
    status,
    error_message,
    records_failed
FROM bronze.bz_Audit_Log
WHERE batch_id = '<specific_batch_id>'
    AND status = 'FAILED'
ORDER BY load_timestamp;
```
