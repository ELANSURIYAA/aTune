Option 1: Conceptual Data Model Development Workflow

1. Agents to consider:
   Fabric Model Conceptual → Fabric Model Data Constraints

2. What we will get:
   • A complete conceptual data model for credit card reporting that includes all core business entities (Application, Campaign, Credit Risk, Fraud Screening, Transaction, Funnel Stage) with their attributes, relationships, and primary keys, derived directly from the requirements document and validated against the five reporting objectives (Application Funnel Analysis, Campaign Performance, Credit Risk Assessment, Fraud Detection, Transaction Analysis).
   
   • A comprehensive set of business rules, data constraints, and validation logic for each entity and attribute in the conceptual model, ensuring that all regulatory compliance requirements (PII handling, data retention policies, audit trail specifications, security access rules) are documented and enforced at the conceptual level before downstream implementation.

   • Dedicated entity and attribute modeling logic that focuses exclusively on identifying core entities, defining attributes with data types and cardinality, establishing relationships with referential integrity rules, and resolving entity ambiguity through structured escalation, which means the conceptual agent must handle the full spectrum of modeling tasks without specialized sub-processes for attribute mapping, relationship validation, or ambiguity resolution, potentially leading to incomplete attribute definitions or unclear relationships.
   
   • Specialized business rules and compliance extraction capability that systematically identifies, categorizes, and validates all business logic, security access rules, data retention policies, and audit trail requirements as distinct artifacts separate from entity definitions, which means compliance and audit requirements may be embedded within entity documentation rather than maintained as a separate, auditable catalog, reducing visibility for compliance officers and auditors and increasing the risk of overlooking critical regulatory requirements.

1. Agents to consider:
   Fabric Model Data Constraints → Fabric Model Conceptual

2. What we will get:
   • An upfront validation and business rules extraction phase where all data expectations, constraints, compliance requirements, security access rules, data retention policies, and audit trail specifications are identified and documented before entity modeling begins, ensuring that the conceptual model is built with full awareness of all regulatory and business logic requirements from the start and reducing the risk of compliance gaps discovered late in the modeling process.
   
   • A conceptual data model that is constructed with pre-validated business rules and constraints already defined, allowing the Fabric Model Conceptual agent to focus exclusively on entity identification, attribute definition, and relationship establishment while referencing the pre-approved rules catalog, which streamlines the modeling process and ensures consistency between business logic and entity structure.
   
   • A two-phase workflow that separates concerns between business logic validation (handled by Data Constraints agent) and structural modeling (handled by Conceptual agent), providing clearer accountability, better traceability between requirements and model components, and a more auditable process where compliance and modeling decisions are documented in distinct phases with explicit handoff points.

   • Final model synthesis and integration capability that assembles all modeling outputs (entities, attributes, relationships, business rules, compliance artifacts) into a single validated conceptual data model package, cross-references it against data governance standards, performs completeness validation, and produces a structured handoff package for downstream teams, which means the workflow ends with separate outputs from two agents rather than a unified, governance-approved conceptual model ready for immediate use by logical and physical modeling teams.

SECTION 4 — IMPROVEMENTS & OPTIMISATIONS

─────────────────────────────────────────────────────────────
ACCURACY — How well the agent produces correct, on-target
outputs and avoids errors.

| Criteria            | Description                          |
|---------------------|--------------------------------------|
| Self-validation     | Requirements Parsing & Validation Agent must re-evaluate its parsed requirements output against a mandatory completeness checklist (business goals, entities, attributes, relationships, compliance sections, reporting objectives) and verify that every section is present and unambiguous before marking the document as validated; Fabric Model Conceptual must cross-reference every entity and relationship in its output against the validated requirements to confirm each has explicit traceability to a requirement statement; Fabric Model Data Constraints must validate that every entity containing PII or sensitive data has documented security access rules, retention policies, and audit trail specifications before finalizing the rules catalog; Model Synthesis & Integration Agent must validate 100% requirement coverage by confirming every requirement statement maps to at least one model component (entity, attribute, relationship, or rule) and every model component traces back to a requirement. |
| Boundary definition | Requirements Parsing & Validation Agent must NOT alter, paraphrase, or interpret the source requirements text—it must extract and structure only what is explicitly stated and escalate any ambiguity rather than making assumptions; Fabric Model Conceptual must NOT create entities, attributes, or relationships that are not explicitly stated or clearly implied in the validated requirements—it must escalate to data architect if additional entities are needed; Fabric Model Data Constraints must NOT invent business rules or compliance requirements—it must extract only what is documented in requirements and escalate to compliance officer if required controls are missing; Model Synthesis & Integration Agent must NOT approve or hand off an incomplete model—it must escalate to project lead if any validation check fails or any requirement is unmapped. |
| Known issues log    | Requirements Parsing & Validation Agent must log all instances of missing sections, ambiguous requirement statements, conflicting requirements, and unresolvable gaps with escalation status and resolution outcome to build a pattern library of common requirements quality issues; Fabric Model Conceptual must log all cases of overlapping entity definitions, unclear relationships, missing primary keys, and attribute definition ambiguities to identify recurring modeling challenges; Fabric Model Data Constraints must log all instances of missing compliance requirements, conflicting business rules, and incomplete audit trail specifications to track compliance risk patterns; Model Synthesis & Integration Agent must log all integration failures, governance validation errors, and incomplete model components to identify systemic quality issues across the modeling pipeline. |

EFFICIENCY — How well the agent produces output that is most
optimised — same correctness in fewer steps or words.

| Criteria                  | Description                    |
|---------------------------|--------------------------------|
| Best practices            | Requirements Parsing & Validation Agent should be provided with 3-5 labelled examples of well-structured, complete requirements documents (showing all mandatory sections present and clearly written) and 2-3 counter-examples of incomplete or ambiguous documents (showing missing sections, unclear statements, conflicting requirements) to learn the expected requirements quality standard and reduce false-positive escalations; Fabric Model Conceptual should receive 5-7 examples of correctly modeled entities with complete attribute definitions, proper primary key selection, and clear relationship cardinality, plus 2-3 counter-examples showing common errors (missing attributes, unclear relationships, invented entities) to learn the expected entity modeling standard; Fabric Model Data Constraints should be given 3-5 examples of complete business rules catalogs with proper compliance documentation (security rules, retention policies, audit trails) and 2-3 counter-examples showing incomplete or generic compliance statements to learn the expected rules documentation standard; Model Synthesis & Integration Agent should receive 2-3 examples of complete, governance-approved handoff packages showing the expected structure, validation reports, and traceability documentation. |
| Defined output structure  | Requirements Parsing & Validation Agent must produce output in a fixed JSON schema with sections: {parsed_requirements: {business_goals: [], entities: [], attributes: [], relationships: [], compliance: [], reporting_objectives: []}, validation_status: {completeness_score: float, missing_sections: [], ambiguous_statements: [], escalations: []}} to prevent verbose narrative output; Fabric Model Conceptual must produce output in a structured ER diagram format (JSON or XML) with schema: {entities: [{name, primary_key, attributes: [{name, data_type, nullable}], relationships: [{target_entity, cardinality, foreign_key}]}]} to ensure consistent, parseable entity definitions; Fabric Model Data Constraints must produce output in a fixed rules catalog schema: {business_rules: [{rule_id, entity, rule_text, validation_logic}], compliance: [{entity, pii_classification, security_rules, retention_policy, audit_trail}]} to prevent wandering compliance narratives; Model Synthesis & Integration Agent must produce output in a fixed handoff package schema: {conceptual_model: {entities, attributes, relationships}, rules_catalog, compliance_documentation, validation_report: {requirement_coverage: [], governance_status}, traceability_matrix}. |
| Define task breakdown     | Requirements Parsing & Validation Agent should decompose its work into discrete subtasks: (1) Extract document sections, (2) Validate section completeness against checklist, (3) Identify ambiguous statements, (4) Generate escalation report, (5) Produce validated requirements JSON—each subtask with explicit success criteria to constrain reasoning overhead; Fabric Model Conceptual should break down entity modeling into: (1) Identify candidate entities from requirements, (2) Define attributes for each entity, (3) Select primary keys, (4) Establish relationships with cardinality, (5) Validate entity completeness, (6) Generate ER diagram—each step producing intermediate output for validation; Fabric Model Data Constraints should decompose rules extraction into: (1) Extract business rules by entity, (2) Identify PII and sensitive data, (3) Document security access rules, (4) Define retention policies, (5) Specify audit trail requirements, (6) Validate compliance completeness—each subtask with explicit compliance checklist; Model Synthesis & Integration Agent should break down integration into: (1) Assemble entity definitions, (2) Merge rules catalog, (3) Validate requirement coverage, (4) Integrate with governance system, (5) Generate validation report, (6) Produce handoff package—each step with explicit validation gate. |

| Hybrid model                   | Requirements Parsing & Validation Agent: Use regex and rule-based parsers to extract document sections, table of contents, and section headers (deterministic extraction), invoking LLM only for semantic validation of section content completeness and ambiguity detection; Fabric Model Conceptual: Use rule-based logic to validate entity naming conventions, primary key format, and attribute data type syntax (deterministic checks), invoking LLM only for entity identification, relationship reasoning, and ambiguity resolution; Fabric Model Data Constraints: Use rule-based compliance checklist validation to verify presence of required compliance sections (security, retention, audit) for each PII entity (deterministic check), invoking LLM only for business rule extraction and compliance requirement interpretation; Model Synthesis & Integration Agent: Use deterministic cross-reference logic to validate requirement-to-model traceability (every requirement ID appears in at least one model component), invoking LLM only for governance integration, validation report generation, and complex error resolution. |
