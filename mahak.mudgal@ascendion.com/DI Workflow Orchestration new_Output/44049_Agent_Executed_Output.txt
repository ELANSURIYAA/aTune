| Criteria            | Description                          |

| Self-validation     | Requirements Parsing & Validation Agent must re-validate its parsed requirements output by cross-referencing the extracted structured requirements against the original requirements document to ensure no sections were omitted, misinterpreted, or incorrectly categorized, and must calculate a completeness score (percentage of required sections successfully parsed) before passing output to downstream agents. Fabric Model Conceptual must re-validate its entity definitions by checking that every entity has a clear business justification traceable to the requirements, that all mandatory attributes are present, and that all relationships are bidirectional and logically consistent. Entity & Attribute Modeling Agent must re-validate its attribute specifications by confirming that every attribute has a defined data type, business rule, and compliance classification, and that no attributes are duplicated or contradictory across entities. Business Rules & Compliance Agent must re-validate its extracted business rules by cross-referencing them against the compliance standards repository to ensure no regulatory requirements are missed and no conflicting rules are present. Model Synthesis & Integration Agent must re-validate the final conceptual model by running a comprehensive coverage check that confirms 100% of requirements are addressed, all entities are integrated, and all governance validations have passed before approving handoff. |
| Boundary definition | Requirements Parsing & Validation Agent must NOT alter, rephrase, or interpret the original requirements text—it must extract and structure only what is explicitly stated, and must NOT proceed with modeling if any critical section is missing or ambiguous without stakeholder escalation. Fabric Model Conceptual must NOT create entities, attributes, or relationships that are not supported by the requirements document, must NOT assume business processes or data flows that are not explicitly described, and must NOT proceed with modeling if entity definitions are unclear without escalating to domain experts. Entity & Attribute Modeling Agent must NOT invent attributes or relationships to "complete" an entity definition, must NOT assume data types or constraints that are not specified in the requirements, and must NOT resolve ambiguity through inference—all uncertain definitions must be escalated. Business Rules & Compliance Agent must NOT omit or simplify compliance requirements to make them easier to implement, must NOT invent business rules to fill perceived gaps, and must NOT approve incomplete compliance documentation without escalation to compliance officers. Model Synthesis & Integration Agent must NOT hand off an incomplete or unvalidated conceptual model, must NOT bypass governance integration steps to accelerate delivery, and must NOT proceed with handoff if any validation check fails without stakeholder approval. |
| Known issues log    | Requirements Parsing & Validation Agent must log all instances of missing sections, ambiguous requirement statements, conflicting business goals, and incomplete entity or attribute specifications encountered during parsing, so that future runs can flag these patterns early and provide targeted guidance to requirements authors. Fabric Model Conceptual must log all cases where entity definitions were unclear, where multiple valid entity interpretations existed, where relationships could not be established due to missing information, and where PII classification was uncertain, so that these patterns can be proactively flagged in future modeling efforts. Entity & Attribute Modeling Agent must log all ambiguity escalations, all cases where attributes were missing or incomplete, all relationship conflicts, and all instances where confidence scores fell below 0.8, so that common ambiguity patterns can be identified and addressed through improved requirements templates or stakeholder training. Business Rules & Compliance Agent must log all instances of conflicting business rules, missing compliance requirements, incomplete audit trail specifications, and cases where regulatory standards were unclear or evolving, so that these risks can be proactively managed in future projects. Model Synthesis & Integration Agent must log all integration failures, all governance validation errors, all cases where model coverage was incomplete, and all handoff delays caused by missing artifacts, so that these failure patterns can be addressed through improved upstream agent outputs and validation checkpoints. |

| Criteria                       | Description               |
|--------------------------------|---------------------------|
| Low cost model routing         | Requirements Parsing & Validation Agent: Use GPT-4o for initial document parsing and ambiguity detection (complex reasoning required), but route completeness validation, section extraction, and structured output generation to GPT-4o-mini (rule-based checks). Fabric Model Conceptual: Use Claude Sonnet 3.5 for entity identification and relationship establishment (requires domain understanding and structured reasoning), but route attribute data type assignment, PII classification (based on predefined rules), and ER diagram generation to GPT-4o-mini (deterministic tasks). Entity & Attribute Modeling Agent: Use GPT-4o for ambiguity resolution and relationship validation (requires complex reasoning), but route attribute specification expansion, compliance classification (rule-based lookup), and output formatting to GPT-4o-mini. Business Rules & Compliance Agent: Use Claude Sonnet 3.5 for business rule extraction and conflict detection (requires precise rule interpretation), but route compliance checklist validation, audit trail specification generation, and output formatting to GPT-4o-mini. Model Synthesis & Integration Agent: Use GPT-4o-mini for all subtasks (model assembly, validation checks, report generation, handoff package creation) as these are primarily deterministic, rule-based operations that do not require advanced reasoning, enabling significant cost savings without sacrificing accuracy. |
| Token size reduction           | Requirements Parsing & Validation Agent: Implement chunked processing for large requirements documents—split document into sections (business goals, entities, attributes, relationships, business rules, compliance) and process each section independently, then merge results, reducing context window from 50,000 tokens to ~5,000 tokens per chunk. Fabric Model Conceptual: Receive only the parsed, structured requirements (not the full original document) from Requirements Parsing & Validation Agent, reducing input context from 50,000 tokens to ~10,000 tokens; cache entity definitions and reuse them across relationship establishment subtasks to avoid redundant context loading. Entity & Attribute Modeling Agent: Receive only entity definitions and attribute candidates (not full requirements or parsed data) from Fabric Model Conceptual, reducing input context to ~8,000 tokens; process each entity independently and merge results to enable parallel processing with minimal context per call. Business Rules & Compliance Agent: Receive only the business rules and compliance sections from parsed requirements (not full document), reducing input context to ~5,000 tokens; cache compliance standards repository and reuse across rule validation subtasks to avoid repeated loading of reference data. Model Synthesis & Integration Agent: Receive only final entity definitions, business rules, and compliance artifacts (not intermediate outputs or full requirements), reducing input context to ~15,000 tokens; use structured JSON inputs to minimize verbose text and enable efficient parsing and assembly. |
| Hybrid model                   | Requirements Parsing & Validation Agent: Use regex and rule-based parsers to extract section headers, table structures, and list items from the requirements document (deterministic task), and invoke the LLM only for ambiguity detection, semantic validation, and escalation log generation (non-deterministic tasks), reducing LLM calls by ~40%. Fabric Model Conceptual: Use rule-based lookups to assign PII flags and compliance classifications to attributes based on predefined taxonomies (deterministic task), and invoke the LLM only for entity identification, relationship establishment, and business justification generation (non-deterministic tasks), reducing LLM calls by ~30%. Entity & Attribute Modeling Agent: Use rule-based validation to check attribute completeness (all mandatory fields present), data type consistency, and relationship cardinality correctness (deterministic tasks), and invoke the LLM only for ambiguity resolution and relationship validation (non-deterministic tasks), reducing LLM calls by ~35%. Business Rules & Compliance Agent: Use rule-based checklist validation to confirm all required compliance sections are present and all audit trail requirements are documented (deterministic tasks), and invoke the LLM only for business rule extraction, conflict detection, and compliance interpretation (non-deterministic tasks), reducing LLM calls by ~40%. Model Synthesis & Integration Agent: Use rule-based assembly logic to merge entity definitions, business rules, and compliance artifacts into the final model structure (deterministic task), use API calls to integrate with data governance system (no LLM required), and invoke the LLM only for validation report generation and handoff package narrative (non-deterministic tasks), reducing LLM calls by ~50%. |
